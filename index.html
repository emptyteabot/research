Executive Summary

一句话结论

  • AI 开发工具正从代码补全，快速进化为可规划、可执行、可测试的开发智能体。
  • 主流产品都在强调自动化，但同样都在强调人工评审、测试、回滚与治理。
  • 真正的机会，不在“再多写一点代码”，而在“把 AI 产出变成团队能接受的交付单元”。
  • 因此我选择做面向产品经理与工程经理的 Spec-driven 交付控制层。
可追溯 Spec 可审计 Patch 可验证 Evidence
Why now

市场正在往前跑，但“可控交付”仍然空着。

过去两年，行业的演进路径很清晰：从助手层，到 Agentic IDE，再到自治工程师。 但每走一步，治理成本都会同步上升。我的判断是，下一个产品机会在自动化和可控性之间的平衡层。

01

助手层

Copilot、灵码、快码这类产品解决“写得更快”，但不直接承担完整交付责任。

02

Agentic IDE

Cursor、Windsurf、TRAE 把能力推进到规划、多文件改动、终端执行与工作流复用。

03

自治工程师

Devin、Replit Agent 把“任务委派给 AI”做成产品核心，强调更长任务链与并行执行。

04

控制平面

真正缺的是把需求、计划、变更、测试证据与 PR 审批串成可审计交付资产。

Research landscape

8 个代表性产品，指向同一个趋势。

这些产品的共性不是“谁更会补全”，而是都开始把 AI 放进真实的工程工作流里。 差别在于，它们把边界画在 IDE、平台、还是自治代理。

Structural pains

为什么“自动化”做得越强，越需要控制层？

静默失败

AI 生成代码可以语法正确，却在业务边界上悄悄出错；没有证据链，就很难在上线前截住。

Day 0 与 Day 2 断裂

行业很擅长“几分钟做出一个 demo”，却很少真正解决上线后的监控、修复、重构与反馈闭环。

上下文退化

当代码库变大、依赖变深、任务变长，智能体更容易丢失上下文，进入改一处崩三处的死循环。

治理边界模糊

谁能放开终端？谁能触发部署？谁来确认变更符合 PRD？这些问题不解决，AI 很难真正进入生产流。

My bet

我选择切入：Spec-driven + 可审计的 PRD → Plan → Patch 工作流层。

为什么不是再做一个 IDE？

因为主流产品已经证明：AI 会写代码并不稀缺，真正稀缺的是 “团队相信这次改动可以被合并”。

  • 把 PRD 变成可追溯 Spec，而不是一句模糊 prompt
  • 把改动变成带风险标签的 Patch，而不是黑盒输出
  • 把正确性变成测试、扫描、运行截图等 Evidence，而不是“看起来应该没问题”
SpecFlow Value

面向 PM / Tech Lead / 工程经理的交付控制台

PRD Spec Plan Patch Evidence PR

用户不需要理解每个模型，也不需要盯着 AI 写每一行代码；他们只需要知道： 需求是否清晰、计划是否合理、改动是否可回滚、证据是否齐全、这次 PR 能不能放心 merge。

Interactive MVP

SpecFlow：一个把“会写代码”变成“会交付”的最小可行原型。

下面这个交互区模拟的是 SpecFlow 的核心工作台：输入一个需求场景， 系统自动生成 Spec、任务计划、Patch 摘要、验证证据与 PR 说明，并支持导出成 Markdown 交付件。

演示场景
这个 MVP 想证明什么
  • AI 产出的第一层对象不再是代码，而是 Spec
  • 计划可编辑、变更可审计、证据可追溯
  • PM 与工程经理也能直接理解交付状态
  • 最后输出的是可 merge 的 PR，而不是一堆零碎对话
